1 소개
이 랩에서는 C 프로그램을 위한 동적 저장소 할당기, 즉 `malloc`, `free`, `realloc` 루틴의 여러분만의 버전을 작성하게 됩니다. 설계 공간을 창의적으로 탐색하여, 정확하고 효율적이며 빠른 할당기를 구현해 보길 권장합니다.

4 랩 작업 방법
여러분의 동적 저장소 할당기는 다음 네 함수로 구성되며, 이들은 `mm.h`에 선언되어 있고 `mm.c`에 정의됩니다.

`int mm_init(void);`
`void *mm_malloc(size_t size);`
`void mm_free(void *ptr);`
`void *mm_realloc(void *ptr, size_t size);`

배포된 `mm.c`는 우리가 생각할 수 있는 가장 단순하지만 기능적으로는 올바른 `malloc` 패키지를 구현해 둔 상태입니다. 이를 출발점으로 삼아 위 함수들(및 필요하다면 `static` 보조 함수들)을 다음 의미론을 만족하도록 수정하세요.

- `mm_init`:
  - 응용 프로그램(즉, 여러분 구현을 평가하는 trace 기반 드라이버)은 `mm_malloc`, `mm_realloc`, `mm_free`를 호출하기 전에 `mm_init`을 호출하여 초기 힙 영역 할당 등 필요한 초기화를 수행합니다.
  - 초기화에 문제가 있으면 `-1`, 아니면 `0`을 반환해야 합니다.

- `mm_malloc`:
  - `mm_malloc`은 최소 `size` 바이트 페이로드를 갖는 할당 블록의 포인터를 반환합니다.
  - 할당 블록 전체는 힙 영역 내부에 있어야 하며 다른 할당 블록과 겹치면 안 됩니다.
  - 채점 시 표준 C 라이브러리(`libc`)의 `malloc`과 비교합니다.
  - `libc malloc`은 8바이트 정렬 포인터를 반환하므로, 여러분 구현도 항상 8바이트 정렬 포인터를 반환해야 합니다.

- `mm_free`:
  - `mm_free`는 `ptr`이 가리키는 블록을 해제하며 반환값은 없습니다.
  - 이 함수는 전달된 포인터(`ptr`)가 이전 `mm_malloc` 또는 `mm_realloc` 호출로 반환된 값이고, 아직 해제되지 않았을 때만 동작이 보장됩니다.

- `mm_realloc`:
  - `mm_realloc`은 최소 `size` 바이트 할당 영역의 포인터를 반환하며 다음 제약을 따릅니다.
  - `ptr`이 `NULL`이면, 호출은 `mm_malloc(size)`와 동일합니다.
  - `size`가 0이면, 호출은 `mm_free(ptr)`와 동일합니다.
  - `ptr`이 `NULL`이 아니면, 반드시 이전 `mm_malloc` 또는 `mm_realloc`이 반환한 값이어야 합니다.
  - `mm_realloc` 호출은 `ptr`이 가리키는 기존 블록(old block)의 크기를 `size` 바이트로 변경하고 새 블록의 주소를 반환합니다.
  - 새 블록의 주소는 구현 방식, 기존 블록의 내부 단편화, 요청 크기에 따라 기존 주소와 같을 수도 있고 다를 수도 있습니다.
  - 새 블록의 내용은 기존 `ptr` 블록 내용 중 old/new 크기 최솟값 범위까지 동일해야 하며, 그 밖은 초기화되지 않습니다.
    - 예: 기존 블록 8바이트, 새 블록 12바이트라면 새 블록의 처음 8바이트는 기존 처음 8바이트와 동일하고, 마지막 4바이트는 미초기화입니다.
    - 예: 기존 블록 8바이트, 새 블록 4바이트라면 새 블록 내용은 기존 처음 4바이트와 동일합니다.

이 의미론은 `libc`의 `malloc`, `realloc`, `free` 의미론과 동일합니다. 자세한 문서는 셸에서 `man malloc`을 참고하세요.

5 힙 일관성 검사기(Heap Consistency Checker)
동적 메모리 할당기는 정확하고 효율적으로 구현하기 매우 까다롭습니다. 타입 정보가 없는 포인터 조작이 많아 특히 정확성 확보가 어렵습니다. 힙을 스캔해 일관성을 점검하는 힙 검사기를 작성하면 큰 도움이 됩니다.

검사 항목 예시는 다음과 같습니다.
- free 리스트의 모든 블록이 실제로 free 상태로 표시되어 있는가?
- 서로 인접한 free 블록이 coalescing되지 않고 남아 있지는 않은가?
- 모든 free 블록이 실제로 free 리스트에 들어 있는가?
- free 리스트 포인터가 유효한 free 블록을 가리키는가?
- 할당된 블록끼리 겹치는 경우가 있는가?
- 힙 블록 내부 포인터들이 유효한 힙 주소를 가리키는가?

여러분의 힙 검사기는 `mm.c`의 `int mm_check(void)` 함수로 구성됩니다. 적절하다고 판단하는 불변식/일관성 조건을 검사하면 됩니다. 힙이 일관적일 때에만 0이 아닌 값을 반환합니다.

위 예시를 모두 검사해야 하는 것은 아니며, 나열된 것 외의 검사도 자유롭게 추가할 수 있습니다. `mm_check` 실패 시 에러 메시지를 출력하도록 권장합니다.

이 검사기는 개발 중 디버깅용입니다. `mm.c` 제출 전에는 처리량 저하를 피하기 위해 `mm_check` 호출을 제거하세요. `mm_check` 함수의 스타일도 평가 대상이므로, 무엇을 검사하는지 주석과 문서화를 충분히 해두세요.

6 지원 루틴
`memlib.c` 패키지는 여러분의 동적 메모리 할당기를 위한 메모리 시스템을 시뮬레이션합니다. `memlib.c`에서 다음 함수를 사용할 수 있습니다.

- `void *mem_sbrk(int incr)`:
  - 힙을 `incr` 바이트만큼 확장합니다. `incr`은 양의 0이 아닌 정수여야 합니다.
  - 새로 할당된 힙 영역의 첫 바이트를 가리키는 일반 포인터를 반환합니다.
  - 의미론은 Unix `sbrk`와 동일하지만, `mem_sbrk`는 양의 0이 아닌 정수 인자만 받습니다.

- `void *mem_heap_lo(void)`:
  - 힙의 첫 바이트를 가리키는 일반 포인터를 반환합니다.

- `void *mem_heap_hi(void)`:
  - 힙의 마지막 바이트를 가리키는 일반 포인터를 반환합니다.

- `size_t mem_heapsize(void)`:
  - 현재 힙 크기(바이트)를 반환합니다.

- `size_t mem_pagesize(void)`:
  - 시스템 페이지 크기를 반환합니다(Linux에서는 4KB).

7 Trace 기반 드라이버 프로그램
`malloclab-handout.tar`에 포함된 드라이버 `mdriver.c`는 여러분의 `mm.c` 패키지를 정확성, 공간 활용도, 처리량 관점에서 테스트합니다. 드라이버는 동일한 배포본에 들어 있는 trace 파일 집합으로 제어됩니다. 각 trace 파일에는 allocate/reallocate/free 지시 시퀀스가 들어 있으며, 이를 따라 `mm_malloc`, `mm_realloc`, `mm_free`를 순서대로 호출합니다.

드라이버와 trace 파일은 채점 시 그대로 사용됩니다.

`mdriver.c` 명령줄 옵션:
- `-t <tracedir>`: 기본 trace 디렉터리(`config.h` 정의) 대신 `tracedir`에서 trace 파일을 찾습니다.
- `-f <tracefile>`: 기본 trace 세트 대신 특정 `tracefile` 하나로 테스트합니다.
- `-h`: 명령줄 인자 요약을 출력합니다.
- `-l`: 학생 구현과 함께 `libc malloc`도 실행하고 측정합니다.
- `-v`: 상세 출력(압축 테이블 형태로 trace별 성능 분해).
- `-V`: 더 상세한 출력(trace 처리 중 추가 진단 정보 표시). 디버깅 시 어떤 trace에서 실패하는지 찾는 데 유용합니다.

8 프로그래밍 규칙
- `mm.c`의 인터페이스는 변경하면 안 됩니다.
- 메모리 관리 관련 라이브러리 호출/시스템 호출을 사용하면 안 됩니다.
  - 즉, `malloc`, `calloc`, `free`, `realloc`, `sbrk`, `brk` 및 이들의 변형 호출 사용 금지.
- `mm.c`에서 배열/구조체/트리/리스트 같은 전역 또는 `static` 복합 자료구조를 정의할 수 없습니다.
  - 단, 정수/실수/포인터 같은 전역 스칼라 변수는 허용됩니다.
- `libc malloc`과 일관되게 8바이트 경계 정렬 포인터를 반환해야 합니다.
  - 드라이버가 이 요구사항을 검사합니다.

9 평가
규칙을 어기거나 코드에 버그가 있어 드라이버가 크래시하면 0점입니다.
그 외에는 다음 기준으로 채점합니다.

- 정확성 (20점):
  - 드라이버의 정확성 테스트를 모두 통과하면 만점.
  - 각 trace를 통과한 정도에 따라 부분 점수 부여.

- 성능 (35점):
  - 두 지표로 평가합니다.
  - 공간 활용도(space utilization):
    - 드라이버가 실제 사용하는 메모리량(`mm_malloc`/`mm_realloc`으로 할당되고 아직 `mm_free`되지 않은 총량) 대비, 할당기가 사용한 힙 크기의 최대 비율.
    - 최적 비율은 1.
    - 단편화를 줄여 이 비율을 최적값에 가깝게 만들어야 합니다.
  - 처리량(throughput):
    - 초당 처리한 연산 수의 평균.

드라이버는 다음 성능 지수 `P`로 성능을 요약합니다.
`P = wU + (1 - w) * min(1, T / Tlibc)`

- `U`: 공간 활용도
- `T`: 여러분 구현의 처리량
- `Tlibc`: 기본 trace에서 시스템 `libc malloc`의 추정 처리량

기본적으로 `w = 0.6`으로, 처리량보다 공간 활용도에 더 큰 가중치를 둡니다.

메모리와 CPU 모두 비용이 큰 자원이므로, 이 공식을 통해 두 지표를 균형 있게 최적화하도록 유도합니다.
이상적으로는 `P = w + (1 - w) = 1`(100%)에 도달합니다.
각 지표가 기여할 수 있는 최대치는 각각 `w`, `1 - w`이므로, 한쪽만 극단적으로 최적화하지 말고 균형을 맞춰야 좋은 점수를 얻을 수 있습니다.

- 스타일 (10점):
  - 코드는 함수 단위로 잘 분해하고 전역 변수는 가능한 적게 사용하세요.
  - 코드 시작 부분에 다음을 설명하는 헤더 주석을 넣으세요.
    - free/allocated 블록 구조
    - free 리스트 조직 방식
    - 할당기가 free 리스트를 조작하는 방법
  - 각 함수 앞에는 함수가 무엇을 하는지 설명하는 헤더 주석을 두세요.
  - `mm_check` 검사기는 충분히 철저하고 문서화되어야 합니다.
  - 힙 검사기가 좋으면 5점, 프로그램 구조/주석이 좋으면 5점을 받습니다.

주석:
`Tlibc` 값은 드라이버 내 상수(600 Kops/s)이며, 강의 담당자가 프로그램 구성 시 설정한 값입니다.

11 힌트
- `mdriver -f` 옵션을 활용하세요.
  - 초기 개발 단계에서는 작은 trace 파일을 쓰면 디버깅/테스트가 쉬워집니다.
  - 이를 위해 `short1-bal.rep`, `short2-bal.rep` 두 trace 파일이 포함되어 있습니다.

- `mdriver -v`, `-V` 옵션을 활용하세요.
  - `-v`는 trace 파일별 상세 요약을 제공합니다.
  - `-V`는 각 trace 파일을 읽는 시점도 알려주므로, 오류가 나는 trace를 분리하는 데 유용합니다.

- `gcc -g`로 컴파일하고 디버거를 사용하세요.
  - 디버거는 경계 밖 메모리 접근 문제를 분리하고 식별하는 데 큰 도움이 됩니다.

- 교재의 `malloc` 구현을 한 줄씩 완전히 이해하세요.
  - 교재에는 implicit free list 기반 단순 할당기 예제가 자세히 설명되어 있습니다.
  - 이를 출발점으로 삼되, 단순 implicit list 할당기를 완전히 이해하기 전에는 구현을 시작하지 마세요.

- 포인터 산술은 C 전처리 매크로로 캡슐화하세요.
  - 메모리 관리 코드의 포인터 산술은 캐스팅이 많아 복잡하고 오류가 나기 쉽습니다.
  - 포인터 연산용 매크로를 작성하면 복잡도를 크게 줄일 수 있습니다.
  - 교재 예시를 참고하세요.

- 단계적으로 구현하세요.
  - 처음 9개 trace는 `malloc`/`free` 요청만 포함합니다.
  - 마지막 2개 trace는 `realloc`, `malloc`, `free`를 포함합니다.
  - 먼저 9개 trace에서 `malloc`/`free`를 정확하고 효율적으로 만든 뒤 `realloc`로 넘어가길 권장합니다.
  - 시작은 기존 `malloc`/`free` 위에 `realloc`을 얹는 방식으로 하고, 높은 성능을 원하면 독립적인 `realloc` 구현이 필요합니다.

- 프로파일러를 사용하세요.
  - 성능 최적화에는 `gprof`가 도움이 될 수 있습니다.

- 일찍 시작하세요.
  - 몇 페이지 코드로도 효율적인 `malloc` 패키지를 작성할 수 있습니다.
  - 하지만 지금까지 작성한 코드 중 가장 어렵고 정교한 축에 속할 가능성이 높습니다.
  - 일찍 시작하고, 행운을 빕니다.
